What needs attention

Point LibraForge at a folder and it'll tell you what's missing, what's ready, and what's already in shape - across tagging, conversion, and organization.

Guide: what the badges and modes mean

Edit modes

  • Full metadata writes everything the match provided: title, author, series, sequence, narrator, year, genre, ASIN.
  • Series only writes only the series name, not the sequence number, and leaves title and other fields untouched. Used when the match confirms series and author confidently but title identity is uncertain, so a possibly-correct local title is never clobbered, and the sequence number is left alone since this mode exists precisely for cases where that number isn't trustworthy yet.

Write modes

  • Smart: only writes if the current embedded tags differ from the match.
  • Fill missing fields only: never touches a field that already has a value, only fills blanks.
  • Overwrite: always writes every matched field, unconditionally.
How does Smart mode decide what counts as "already matching"?

Smart mode compares each field the match would write against what's already embedded in the file: title, author, series, genre, and ASIN, plus sequence, narrator, and year when the edit mode is Full. It only marks the book Smart-skipped (nothing written) when every one of those checked fields already matches. If even one differs, the file is written normally.

Force reprocess only bypasses the "already processed" check, so the book is re-searched and re-matched from scratch. It does not override what the write mode decides once a fresh match is found:

  • Smart + Force: re-matches from scratch, but if the fresh match's fields are identical to what's already embedded, the book still shows Smart-skipped. Force restarts the search, not the write.
  • Fill missing + Force: still only fills currently-blank fields either way, a no-op if nothing's blank.
  • Overwrite + Force: the only combination that reliably rewrites everything, since Overwrite always writes the full fresh match regardless of what's already there.

Suggested Workflow

Optional first step: stage new books in one folder, e.g. _unorganized. Not strictly necessary, but keeps newly-added books separate from your organized library.

  1. Dry run with backupMetadata Forge, Apply off, Back up and cache on. Pre-populates the cache before anything is written.
  2. Review report, generate Suspect ReportCheck Matched and Manual Review, then run Suspect Report to catch likely false positives.
  3. Fix bad matches manuallyResolve anything in Manual Review by hand.
  4. ApplyWrites the matched metadata for real.
  5. Organizer: Index library (one time only)Refreshes the destination-structure cache before the first real run.
  6. Organizer: dry run + consolidatePreviews the planned moves. Consolidation should always be checked as a full-root dry run first.
  7. Review the plan, generate Suspect ReportCatch bad destination paths and move anything wrong manually.
  8. ApplyCommits the previewed moves.

metadata.json

An Audiobookshelf-native metadata.json sidecar is written for every matched book. It's LibraForge's one guaranteed, universal output, independent of audio format or whether embedded-tag writing is even possible or desired (--metadata-json-only skips tags entirely; metadata.json is written either way).

On repeat runs it follows the same write-mode skip logic as tags, so there is no needless rewrite once nothing's changed. But it is not the same file as libraforge.json, which sits right alongside it and holds LibraForge's own internal tracking: whether and how a book was processed, cached duration data, and more.

Because they're separate files: Audiobookshelf's own Store metadata with item server setting only controls where Audiobookshelf writes its own copy of metadata.json: next to the book if enabled, or internally under /metadata/items/ if not. Either way it never touches libraforge.json, so LibraForge's own tracking is unaffected.

Recommended setup: keep Store metadata with item OFF, so Audiobookshelf never overwrites the metadata.json LibraForge wrote. This still works because Audiobookshelf's scanner reads whatever metadata.json is already sitting in the book's folder regardless of that setting, it only governs Audiobookshelf's own writes, not what the scanner reads. Then set the library's Scanner metadata priority to Audiobookshelf metadata file highest, Audio file meta tags second, Folder structure lowest, so LibraForge's matches win. See Audiobookshelf's own Book Scanner guide for the exact settings.

Report types

What each tool produces, and where to find it.

  • Metadata Forge report: per-book match/write outcome (the badges above), downloadable after a run.
  • M4B Tool report: conversion/merge outcomes.
  • Folder Forge report: planned/applied moves.
  • Library Downloader report: per-book download outcome.
  • Suspect Report: generate a report based on the last run. Targets full matches passing certain triggers, to simplify finding a bad full write manually. Note: some false positives will always show up, the idea is to significantly narrow down manually reviewing a full run report, without affecting the actual matching capability.
